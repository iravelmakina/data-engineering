# Assignment: Hourly Support Call Enrichment Pipeline (Airflow/MySQL/JSON/DuckDB)

## Business reason
Support teams need near-real-time visibility into call quality and context.  
This pipeline enriches raw support call logs with telephony metadata and an LLM-style call summary, then loads a clean analytical table for reporting, QA, and monitoring.

---

## Assignment goal
Build an **Apache Airflow DAG** that runs **every hour** and incrementally processes support call data:
- Reads new calls from MySQL
- Enriches them with telephony data from JSON files (mock API)
- Writes the final enriched dataset into DuckDB

## DAG structure (recommended best-practice design)

### 1. `detect_new_calls`
- Query MySQL for calls with: call_time > last_loaded_call_time
- Store and read the watermark using:
  - Airflow Variables **or**
  - A metadata table in DuckDB
- Output list of new `call_id`s (via XCom)

---

### 2. `load_telephony_details`
- For new call IDs:
  - Read corresponding JSON files
  - Parse and validate schema
  - Output parsed telephony records (XCom or staging file)

---

### 3. `transform_and_load_duckdb`
- Join:
  - `calls`
  - `employees`
  - telephony JSON data
  - Insert or upsert into DuckDB
  - Update watermark **only after successful load**

---

## Important implementation notes
- Tasks must be **idempotent**
- Re-running the DAG must **not duplicate data**
- Use primary key (`call_id`) or merge logic in DuckDB
- Prefer hooks/operators instead of raw connections

## Theory questions (Airflow)

1. What is a **DAG** in Apache Airflow and what problem does it solve?
   - A **DAG** (Directed Acyclic Graph) is a collection of tasks organized with directional dependencies. It solves the problem of workflow management by defining the order of execution, handling retries, and providing a visual representation of complex data pipelines.

2. What is a **task** in Airflow and how is it different from a DAG?
   - A **task** is a single unit of work within a DAG (e.g., running a Python function or a SQL query). While a DAG is the container/workflow, the task is the actual execution step.

3. What is an **Operator** in Airflow?  
   Give one example of an operator that could be used in this assignment.
   - An **Operator** is a template for a predefined task. For example, the `PythonOperator` is used in this assignment to execute the custom logic for detecting calls and loading JSON files.

4. What is the difference between **Operators** and **Hooks** in Airflow?
   - **Operators** define "what" to do (the task), while **Hooks** provide a high-level interface to external systems (the "how" to connect). In this DAG, we use `MySqlHook` to interact with the database within a `PythonOperator`.

5. What is **XCom** and when should it be used?  
   Give an example of how XCom is used in this DAG.
   - **XCom** (Cross-Communication) is a mechanism for tasks to share small amounts of data. In this DAG, it is used to pass the list of `call_id`s from the `detect_new_calls` task to the `load_telephony_details` task.

6. What is an **Airflow Connection** and why should credentials not be hardcoded in DAGs?
   - An **Airflow Connection** stores configuration (host, login, password) for external systems in a secure, centralized way. Credentials shouldn't be hardcoded to ensure security, portability, and easier management across environments.

7. What are **Airflow Variables** and when would you use them instead of XCom?
   - **Variables** are global key-value stores used for general configuration (like file paths or environment flags). Unlike XCom, which is task-specific and temporary, Variables are persistent and accessible by any DAG.

8. What is a **cron expression** and how is it used to schedule an Airflow DAG?  
   Give an example of a cron expression for running a DAG every hour.
   - A **cron expression** is a string of five or six fields representing a schedule. In Airflow, it defines the `schedule_interval`. For an hourly run, the expression is `0 * * * *`.

9. What does the **catchup** parameter do in Airflow?  
   Why is `catchup=False` often used for hourly production pipelines?
   - **Catchup** determines whether Airflow should run DAG instances for any missed intervals between the `start_date` and the current time. `catchup=False` is used in production to avoid overwhelming the system with many simultaneous runs when a DAG is first turned on.

10. What does **idempotency** mean in data pipelines and why is it important for Airflow DAGs?
   - **Idempotency** means that running the same pipeline multiple times with the same input results in the same outcome without creating duplicates or errors. It's critical for Airflow so that if a task fails and is retried, it doesn't corrupt the data.
